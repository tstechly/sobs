# SOBS End-to-End Validation Plan

## Purpose

This plan validates the complete SOBS flow with correlated GitHub repo metadata,
release fixtures, OTEL telemetry, RUM activity, CVE enrichment, and agent-flow signals.
It also covers CI-style release metadata (`commitSha`, `buildId`, release metadata JSON)
and JS source-map remapping fixtures.

## Why this matters

A complete test pass should answer all of these at once:

- Can SOBS correlate services and versions to registered repos/releases?
- Are vulnerable dependencies detected and visible in CVE findings?
- Do fixed versions reduce findings and support disposition lifecycle checks?
- Do trace/log/error/metric/RUM signals align by service/version/environment?
- Are agent-flow issue/PR testing surfaces populated and usable?

## Components

### 1) GitHub fixture repos

Created by [scripts/setup_repo_connect_validation.sh](scripts/setup_repo_connect_validation.sh):

- `<prefix>-vuln-fixture`
- `<prefix>-fixed-fixture`
- `<prefix>-agent-playground`

### 2) SOBS release/artifact registration

Performed by [scripts/register_validation_fixtures_to_sobs.sh](scripts/register_validation_fixtures_to_sobs.sh),
which uses [scripts/register_release_artifacts.py](scripts/register_release_artifacts.py).

Each fixture release includes:

- deterministic `commitSha`
- deterministic `buildId`
- release metadata JSON (`ci`, `fixture` fields)
- `js_sourcemap` artifact metadata row (`app.min.js.map`)

### 3) Correlated telemetry generator

Generated by [scripts/generate_validation_telemetry.py](scripts/generate_validation_telemetry.py):

- OTEL traces
- OTEL logs
- direct errors
- OTEL metrics
- RUM events

All emitted with matching `service.name`, `service.version`, and environment.

When `--source-map-dir` is provided, the generator also writes fixture
`app.min.js` + `app.min.js.map` files under:

- `<source-map-dir>/static/<service.name>/app.min.js`
- `<source-map-dir>/static/<service.name>/app.min.js.map`

and emits JS stack frames that should remap when:

- `SOBS_SOURCE_MAP_ENABLE=true`
- `SOBS_SOURCE_MAP_DIR=<source-map-dir>`

### 4) Single-command orchestrator

Use [scripts/run_validation_e2e.sh](scripts/run_validation_e2e.sh) for full workflow orchestration.

## Scenario matrix

| Scenario | Service | Version | Expected signal pattern | Expected enrichment outcome |
|---|---|---|---|---|
| Vulnerable active release | `<prefix>-vuln-fixture` | `0.1.0` | Higher latency, error trace/log + direct error, RUM page/resource events | CVE findings should exist for vulnerable deps |
| Fixed release | `<prefix>-fixed-fixture` | `1.0.0` | Healthy traces/logs, lower p95 metrics, normal RUM | Fewer/no CVE findings compared to vulnerable fixture |
| Agent playground | `<prefix>-agent-playground` | `0.1.0` | Operational telemetry and seeded GitHub issues | Repo health and agent issue workflows are testable |

## Recommended run order

1. Create fixture repos:
   - `./scripts/setup_repo_connect_validation.sh --org <org> --prefix <prefix>`
2. Register repos/releases/artifacts in SOBS:
   - `./scripts/register_validation_fixtures_to_sobs.sh --org <org> --prefix <prefix> --base-url <sobs-url>`
3. Emit correlated telemetry:
   - `python3 ./scripts/generate_validation_telemetry.py --org <org> --prefix <prefix> --base-url <sobs-url> --source-map-dir <map-dir>`
4. Trigger CVE scan:
   - `curl -X POST <sobs-url>/api/enrichment/cve/scan -H 'Content-Type: application/json' -d '{}'`

Or run everything in one command:

- `./scripts/run_validation_e2e.sh --org <org> --prefix <prefix> --base-url <sobs-url>`
- `./scripts/run_validation_e2e.sh --org <org> --prefix <prefix> --base-url <sobs-url> --source-map-dir <map-dir>`

## Assertions checklist

### CVE page

- Vulnerable fixture service/version appears with findings.
- Fixed fixture service/version appears with fewer/no findings.
- Disposition updates persist and behave correctly across rescans.

### Web Traffic page

- RUM events are present for all fixture services.
- URLs/assets include version-correlated hints.

### Traces/Logs/Errors pages

- Shared service names and versions appear across all signal types.
- Vulnerable service has clear error path footprint.

### Repositories settings and repo health

- Each fixture repo is visible and linked.
- Release versions show expected coverage for repo-health context.

### Release metadata and source maps

- Release records contain expected `commitSha`, `buildId`, and fixture `metadata` fields.
- `js_sourcemap` artifact rows exist for fixture releases.
- Error/RUM stack traces include remapped `[mapped]` frames when source maps are enabled.

### Agent flow testing

- Seeded issues and labels exist in agent playground repo.
- Agent-run issue/PR tests have known safe fixture targets.
- Validation must distinguish between:
   - issue creation or issue reuse
   - dedupe suppression / related-link behavior
   - Copilot assignment eligibility
   - active Copilot work already in progress
- Copilot validation should not rely on a plain issue comment mention alone.
- Assignment-oriented tests should enforce conservative limits so only one active Copilot-start request is attempted at a time.

## Planned Agent-Issue Validation Expansion

The agent validation flow is being extended from "create a GitHub issue" to "decide whether to create, reuse, link, or assign".

Expected assertions:
- A new incident with no strong match creates a GitHub issue and a work-item row.
- A semantically duplicate incident reuses the existing issue and increments local occurrence tracking.
- A semantically related incident links to the canonical issue without opening duplicate work.
- Copilot assignment is skipped when an issue already has active work or an open linked PR.
- Copilot assignment requests are globally and per-repo throttled.

## Notes

- Prefer private org/repo ownership for fixture isolation.
- Use a bot/machine account where practical for cleaner contribution history.
- Keep fixture data deterministic to avoid flaky validation outcomes.
