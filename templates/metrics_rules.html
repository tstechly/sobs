Metrics Rules Walkthrough
How Metrics Rules Work (Simple)

SOBS turns raw telemetry into minute-level derived signals, then evaluates rules. When a rule changes state to warning or outlier, you can connect notification rules and agent automation to raise or reuse GitHub issues.

Derived Signals Metrics Rules Notification Rules Auto Issue / Copilot
Threshold

One signal compared to warning and critical thresholds.

Composite

Primary and secondary signals must both be active.

Seasonal

Thresholds vary by hour/day bucket to reduce false positives on periodic traffic.

Example 1: Threshold Rule

Signal: trace error ratio. Warning = 0.08, Critical = 0.12, Min sample = 10.

signal trend warning event outlier event
Trigger example: after sustained rise, one point crosses warning then another crosses critical. A notification rule can send Slack/email, and an agent rule can auto raise or dedupe a GitHub issue when state is warning or critical.
Example 2: Composite Rule

Primary: error ratio. Secondary: trace volume. Both must be active at the same time.

Trigger example: high error ratio alone does not fire if traffic is low; once trace volume threshold is also crossed, composite rule activates. This is useful for reducing low-traffic noise before creating notifications/issues.
Example 3: Seasonal Rule

Signal: request latency. Different thresholds by hour-of-day (UTC), so peak hours can have higher baseline.

seasonal baseline envelope actual series bucket-triggered event
Trigger example: point breaches the active UTC bucket threshold. This usually catches true anomalies while ignoring predictable daily swings. The same event can then feed notification channels and agent issue automation.
